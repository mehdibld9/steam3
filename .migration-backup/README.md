# SteamShare (Account Exchange)

A Steam account marketplace where users share unused Steam libraries, claim accounts using points, and earn XP by participating. Includes a built-in Steam library checker for credential verification.

## Database — Supabase

This app uses **Supabase PostgreSQL** (standard Postgres, works with Drizzle ORM and express-session).

### 1. Create a Supabase project

1. Go to [supabase.com](https://supabase.com) and create a new project.
2. Wait for the database to finish provisioning.

### 2. Get your connection string

In Supabase: **Project Settings → Database → Connection string → URI**

Use one of these (both work with this app):

| Type | When to use |
|------|-------------|
| **Session pooler** (port 5432) | Recommended for Render / long-running servers |
| **Direct connection** (port 5432) | Also fine for a single Render web service |

Do **not** use the **Transaction pooler** (port 6543) — it breaks session storage.

Replace `[YOUR-PASSWORD]` with your database password from project setup.

### 3. Push the schema

```bash
cp .env.example .env
# Paste your Supabase URI into DATABASE_URL

pnpm install --ignore-scripts
pnpm --filter @workspace/db run push
```

This creates all tables. The `session` table for login cookies is created automatically on first server start.

## Deploy to Render

Hosting is on **Render**; the database is on **Supabase**.

1. Push this repo to GitHub.
2. Set up Supabase and run `pnpm --filter @workspace/db run push` locally once (or let Render run it on first deploy — see below).
3. In [Render](https://render.com), click **New → Blueprint** and connect your GitHub repo.
4. When prompted, paste your **Supabase `DATABASE_URL`** as an environment variable.
5. Deploy. The build runs `push-force` to sync the schema automatically.

The web service serves both the React frontend and the Express API from one origin, so login sessions work without cross-origin issues.

### Render environment variables

| Variable | Required | Source |
|----------|----------|--------|
| `DATABASE_URL` | Yes | Supabase → Database → Connection string (URI) |
| `SESSION_SECRET` | Yes | Auto-generated by Blueprint, or set a long random string |
| `NODE_ENV` | Yes | `production` (set by Blueprint) |
| `SKIP_INSTALL_DEPS` | Yes | `true` — lets the build script install deps with pnpm |

### Manual Render setup

| Setting | Value |
|---------|-------|
| Runtime | Node |
| Build Command | `bash render-build.sh` |
| Start Command | `node --enable-source-maps artifacts/api-server/dist/index.mjs` |
| Health Check | `/api/healthz` |

## Deploy to Vercel (frontend + API)

The repo includes a root `server.ts` so Vercel runs the Express API as a serverless function alongside the Vite frontend.

| Setting | Value |
|---------|--------|
| **Root Directory** | *(leave empty — repo root)* |
| **Framework Preset** | Other |
| **Install / Build / Output** | *(leave empty — `vercel.json` sets these)* |

**Do not use `public`** — Vite outputs to `artifacts/steamshare/dist/public`.

### Vercel environment variables (required)

Add these in **Project → Settings → Environment Variables** for Production (and Preview if you test there):

| Variable | Required | Value |
|----------|----------|-------|
| `DATABASE_URL` | **Yes** | Supabase → Database → Connection string (URI). Use **Session pooler** or **Direct** (port 5432). Do **not** use Transaction pooler (6543). |
| `SESSION_SECRET` | **Yes** | Long random string (e.g. `openssl rand -hex 32`) |
| `NODE_ENV` | Yes | `production` |

Without `DATABASE_URL`, every `/api/*` request returns **500** (`FUNCTION_INVOCATION_FAILED`) because the server cannot connect to Postgres.

After the first deploy with `DATABASE_URL` set, run the schema push once locally:

```bash
pnpm --filter @workspace/db run push
```

The `session` table is created automatically on the first API request.

### If API still returns 500

1. Open **Vercel → Project → Logs** and check the function error (often `DATABASE_URL must be set` or a Postgres SSL/connection error).
2. Confirm `DATABASE_URL` is set for the environment you are viewing (Production vs Preview).
3. Redeploy after adding env vars — existing deployments do not pick up new variables until redeployed.
